# 除已讨论内容外，当前项目仍需重点分析的关键点（2026-04-16）

## 文档性质

- 本文是研究阶段的“问题地图”，不是最终方案。
- 目标不是扩张范围，而是识别：在已经讨论了 turn-taking、duck_only、preview/final-ASR、UEPG、slot completeness、dynamic bias/entity catalog 之后，**还有哪些分析点会直接决定语音交互体验上限**。
- 当前阶段仍以：
  - 流畅性
  - 自然性
  - 人性化
  - 智能性
  为主目标。

## 一句话结论

在已经讨论的内容之外，当前项目最值得继续分析的，不是“再换更大的模型”，而是下面这 8 个方向：

1. 端到端时延预算与主观体感映射
2. 播放事实回传与 heard-text 真相链
3. TTS 起播、断句、韵律与人味儿
4. 真实声学环境下的全双工鲁棒性
5. 歧义澄清、低置信处理与错误恢复
6. 高风险动作的分级提交与确认策略
7. 面向真实使用场景的评测体系与数据闭环
8. GPU 资源调度与多模块并行竞争下的实时性稳定性

如果只从“对体验提升的杠杆”排序，我当前最建议优先分析的是：

- `时延预算`
- `播放事实回传`
- `TTS 韵律/断句`
- `评测体系`

## 1. 端到端时延预算与“主观体感”映射

我们已经讨论了：

- preview partial
- early processing
- duck_only
- final-ASR correction

但还没有把它们系统地压成一套 **体验导向的时延预算表**。

### 为什么这很关键

因为系统并不只是“整体快”就行，而是不同节点分别影响不同主观感受：

- `speech_start visible`：决定用户是否感觉“系统在听”
- `first stable preview`：决定用户是否感觉“系统开始懂了”
- `endpoint accept`：决定用户是否感觉“系统知道我说完了”
- `first text draft`：决定用户是否感觉“系统开始思考了”
- `first audio byte / audible syllable`：决定用户是否感觉“系统开始回应了”

### 当前还缺什么分析

- 哪些节点必须压到 100ms / 200ms / 400ms 级
- 哪些节点慢一点用户仍可接受
- 不同场景（问答、命令、纠错、长解释）时延预算是否不同

### 为什么它值得优先

因为没有这张预算表，后续任何优化都会变成“到处加速”，但未必提升用户体感。

## 2. 播放事实回传与 `heard-text` 真相链

我们已经多次讨论：

- 服务侧需要知道用户到底听到了多少
- interruption 之后不能把“生成全文”当成“用户听到全文”

但这仍值得单独深挖，因为它是当前所有“像人”策略的地基之一。

### 为什么这很关键

如果没有真实 playback facts：

- `duck_only` 很难判断自己是否真的起作用了
- `hard_interrupt` 发生后，不知道该从哪里续说
- memory 可能记错为“系统已经说过”，但用户根本没听完
- clarification / correction 会错位

### 当前还缺什么分析

- 端侧最小要回报哪些播放事实？
  - playback started
  - played duration
  - buffer cleared
  - local duck active
  - replay/stop/mute state
- 不同 adapter 能提供多强精度
- 如果只能启发式估计，误差会对体验造成什么后果

### 为什么它值得优先

因为它直接关系到：

- 多轮对话一致性
- interruption 后的人味儿
- 记忆是否可信

## 3. TTS 起播、断句、韵律与“人味儿”

我们已经讨论过 early TTS、首句尽早起播，但还没系统讨论 **“说得像不像人”** 这条线。

### 为什么这很关键

用户感受到的自然性，很大一部分不是由 ASR 决定，而是由：

- 什么时候开口
- 一次说多长
- 在哪里停顿
- 是不是会“抢说”
- 语气是不是死板

### 当前还缺什么分析

- clause / sense-group 切分怎么做
- 什么样的短首句最适合早起播
- 哪些类型回复适合先给 `ack + skeleton`，哪些适合直接给结论
- TTS prosody policy：
  - 确认时
  - 纠错时
  - 安抚时
  - 解释时
  - 提问时
  是否应有不同风格
- 打断后如何恢复语气，而不是像播放器断带

### 为什么它值得优先

因为就算识别再快，如果 TTS 起播生硬、停顿奇怪、语气模板化，用户仍会觉得“不像人”。

## 4. 真实声学环境下的全双工鲁棒性

我们现在讨论了很多“逻辑上”的全双工，但还没充分把它放进真实声学世界里拷问。

### 为什么这很关键

实验室逻辑正确，不代表真实设备上就自然：

- 远场回声
- 双讲
- 背景电视
- 空调噪声
- 手机外放
- 多人说话
- 房间混响

这些都会直接影响：

- duck_only
- hard_interrupt
- device-directedness
- preview quality
- slot completeness

### 当前还缺什么分析

- 真实环境中最常见的失败模式是什么
- 哪些失败是端侧音频前端问题，哪些是服务侧策略问题
- 真正的“近端用户在插话”与“环境声/回声”如何区分到什么程度才够用

### 为什么它重要

因为如果逻辑只在干净样本上成立，项目的研究结论会被高估。

## 5. 歧义澄清、低置信处理与错误恢复

我们目前讨论了：

- 早处理
- final correction
- slot completeness

但还没深挖：**“当系统其实不确定时，应该怎么像人一样处理”**。

### 为什么这很关键

很多助手不自然，不是因为它不会，而是因为它：

- 明明不确定却硬执行
- 明明听错了却不澄清
- 出错后只会重复模板抱歉

### 当前还缺什么分析

- 哪些低置信情况应该：
  - 默默继续听
  - 先澄清
  - 给候选列表
  - 走保守默认
  - 拒绝执行
- 澄清最好发生在：
  - preview 时
  - final 前
  - final 后执行前
- 澄清文案和语气如何不显得机械

### 为什么它重要

因为“人性化”和“智能性”很多时候不来自一次全对，而来自：

- 不确定时处理得是否优雅

## 6. 高风险动作的分级提交与确认策略

我们已经讨论过 `action_risk`，但还没有系统化展开。

### 为什么这很关键

语音系统天然有误听风险，而不同动作的容错率差异极大：

- `明天周几` 回错一次，代价很小
- `删除文件`、`发消息给某人`、`解锁门`，代价完全不同

### 当前还缺什么分析

- 哪些动作属于：
  - 无风险
  - 低风险
  - 中风险
  - 高风险
- 不同风险级别对应：
  - preview-ready
  - draft-ready
  - commit-ready
  的门槛应该如何差异化
- 哪些动作需要二次确认
- 哪些动作可以 silent commit

### 为什么它重要

因为这是“智能”和“可信赖”之间的桥梁。

## 7. 面向真实使用场景的评测体系与数据闭环

这是我认为目前最容易被低估、但其实非常核心的一条。

### 为什么这很关键

如果没有对的评测，后续所有研究都很容易自我感觉良好。

### 当前还缺什么分析

除了传统 CER/WER，还要系统定义：

- first preview latency
- first stable prefix latency
- endpoint accept latency
- first audio latency
- barge-in cutoff latency
- heard-text accuracy
- slot accuracy
- command execution accuracy
- dangerous false accept rate
- correction recovery success rate
- subjective interruption naturalness
- subjective prosody / human-likeness

还需要明确评测集分层：

- 命令型
- 问答型
- 纠错型
- 多轮 follow-up
- 双讲/噪声/远场
- 中英混说
- 长尾实体

### 为什么它值得优先

因为没有这套评测，研究很难真正迭代。

## 8. GPU 资源调度与多模块并行竞争下的实时性稳定性

你已经明确说明：

- 当前机器 GPU 资源充足
- 项目必须在 GPU 环境运行，才能保证实时性

这意味着除了模型能力本身，还必须分析：

- 多模块并行竞争下的延迟抖动

### 为什么这很关键

当前链路里可能同时竞争 GPU 的包括：

- online preview ASR
- final ASR
- VAD / KWS（视实现而定）
- 本地 LLM
- TTS

如果资源调度不好，会出现：

- 平均时延看似不错
- 但 P95/P99 抖动很差
- 某些关键节点偶发卡顿

### 当前还缺什么分析

- 哪些模块适合常驻
- 哪些模块适合 prewarm
- 哪些模块适合批/流混合
- 哪些模块应隔离 GPU / CUDA stream / worker
- 哪类负载下最容易互相抢占

### 为什么它重要

因为“体感流畅”更怕抖动，而不只是怕平均值慢。

## 如果只选下一批最值得分析的 5 个主题

我当前推荐顺序是：

1. 端到端时延预算与主观体感映射
2. 播放事实回传与 `heard-text` 真相链
3. TTS 起播、断句、韵律与人味儿
4. 面向真实使用场景的评测体系与数据闭环
5. 高风险动作的分级提交与确认策略

## 暂时不建议过早投入太多精力的方向

为了避免过度设计，我当前不建议现在就把重点放在：

- 大规模个性化长期建模
- 多模态复杂感知融合
- 重型端侧模型体系
- 大规模在线学习闭环
- 过早设计复杂控制平面 / 权限体系

不是这些不重要，而是当前研究阶段 ROI 低于上面那 8 个方向。

## 一句话总结

除了我们已经讨论的内容，当前项目真正还缺的关键分析点，大多集中在：

- **怎么把系统做得“更像人在说话和听话”**
- **怎么把“已经生成了什么”变成“用户实际听到了什么”**
- **怎么用正确的评测和时延预算去约束整个链路**
